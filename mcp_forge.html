The Problem

Everyone writes the same server, badly

Every team building agentic AI systems in 2026 ends up hand-rolling the same Model Context Protocol server. The protocol itself is solved — but each individual server still gets written from scratch, and the same mistakes appear every time: hallucinated parameters accepted because schemas were not strict, no audit logging, secrets leaking into tool definitions, no structured handling of rate limits or auth failures, no adversarial tests.

Writing one MCP server is annoying. Writing the tenth is a waste of senior engineering time. And the boring infrastructure failures are what actually break agents in production, not the protocol.

The Solution

How the Forge works, end to end

One input — an OpenAPI spec. One output — a complete, deployable, auditable MCP server. Four Bob agents, two human approval gates, one replayable BobShell recipe.

01
Bob Architect mode

Spec ingestion and dependency analysis

Bob reads the OpenAPI spec and builds a full dependency graph — all endpoints, parameters, auth methods, rate limit headers, and schema ambiguities. A lighter Granite model handles parsing; a stronger Claude model reasons about missing or conflicting fields. Output is a structured plan artifact saved to the BobShell context.

02
Human checkpoint

Approval gate — plan review

Before any code is written, the plan is surfaced to the user via Bob's approval checkpoint. The user can modify scope, flag endpoints to exclude, or approve and continue. This gate is not optional — it is a hard stop baked into the BobShell recipe, satisfying enterprise governance requirements.

03
Bob Builder mode

Code generation with security-by-default

A stronger model generates the full MCP server with strict input validation, env-var-based secret handling, structured error responses, and audit logging baked in. Bob's secrets detection layer refuses to embed auth values into tool schemas — they are lifted to environment variables by default. Inline Semgrep scanning runs as a build gate; generation fails if static analysis fails.

04
Bob Tester mode

Adversarial test suite generation and execution

An adversarial test suite is generated and run automatically — not just unit tests, but malformed input tests, injection attempts, and rate limit edge cases. Results surface in Bob's Findings panel. A second human approval gate is offered before the server is marked deployable.

05
Bob Documenter mode

Documentation, packaging, and BobShell recipe

Bob generates README, env variable reference, deployment instructions, and a BobShell recipe file. The recipe is the primary artifact — it encodes the full generation pipeline so when the upstream API changes, the user re-runs one command and gets an updated, diff-able server. The Forge is the artifact; the server is the byproduct.

Risk Register

Honest risks and mitigations

High risk

A funded startup ships this first

The open core model and the compliance layer are your time-to-moat hedge. Generic competitors will not build regulated-industry adapters quickly — they lack the domain expertise. Move to the compliance layer fast.

High risk

IBM Bob lock-in reduces portability

The Forge logic should be Bob-native for the hackathon, but the underlying generation pipeline should be platform-portable. Bob is the accelerator, not the foundation. Design the pipeline to be extractable.

Medium risk

OpenAPI spec quality varies wildly

Invest early in a spec linting and normalization layer. The Forge should improve a bad spec, not fail on it. This is also a differentiation point — competitors will fail on messy real-world specs.

Medium risk

MCP itself is still evolving

Pin to a specific MCP SDK version in each recipe. The regenerability model is your hedge here — when MCP changes, users re-run the recipe and get an updated server automatically. This is a feature, not a bug.

The forge is the artifact. The servers are byproducts.
Implementation Roadmap

Build plan: structure first, layers second

Start with the skeleton that proves the pipeline works end to end. Then add each capability layer one at a time, with a test gate between every milestone before moving forward.

M1
Foundation — Week 1

Skeleton pipeline: OpenAPI in, stub server out

Days 1–5

Build the absolute minimum that proves the concept: a Bob-orchestrated pipeline that reads an OpenAPI spec and emits a syntactically valid, runnable MCP server stub. No security, no tests, no docs yet — just proving the pipeline works.

Set up Bob project structure and BobShell entry point

Create the BobShell recipe file, define the four-agent mode sequence (Architect → Builder → Tester → Documenter), wire up the Orchestrator handoff chain.

Build the spec ingestion module (Architect mode)

Parse YAML and JSON OpenAPI specs into a normalized internal representation. Handle missing fields gracefully — do not crash on imperfect specs. Use Granite model for parsing, output a structured plan JSON artifact.

Implement stub code generator (Builder mode)

Generate a syntactically correct but minimal MCP server in Python. Tool definitions mapped from spec endpoints. No validation logic yet — just the skeleton structure with placeholder handlers.

Wire the first end-to-end run on a real spec

Use the Stripe API OpenAPI spec as the test case. Run the full pipeline once. Verify the output server starts without errors and the tool list matches the spec endpoints.

# BobShell entry point — milestone 1 skeleton forge: input: "./specs/stripe.yaml" agents: - mode: architect # parse spec → plan.json - mode: builder # plan.json → server stub output: "./output/stripe-mcp/"
✓

Milestone 1 test gate — must pass before proceeding

  • Pipeline runs without errors on the Stripe spec
  • Generated server starts and responds to an MCP handshake
  • Tool list in generated server matches endpoints in spec
  • BobShell recipe is re-runnable and produces identical output
M2
Security layer — Week 1–2

Add strict validation, secret handling, and Semgrep gate

Days 4–9

Upgrade the generated server with the security-by-default defaults that hand-rolled servers consistently miss. This is the layer that makes the Forge output materially better than anything a developer would write by hand.

Implement strict input validation from spec schemas

Generate Pydantic models or JSON Schema validators directly from the OpenAPI parameter schemas. Every tool in the generated server must reject inputs that do not match the spec. No hallucinated parameters accepted.

Implement env-var-based secret handling

Wire Bob's secrets detection layer into the Builder mode. Any auth value, API key, or credential found in the spec is automatically lifted to an environment variable reference. The generated server reads from env, never from hardcoded values. Verify via Bob's secrets scan.

Add structured error responses

Generate consistent error response schemas for all tool handlers — rate limit errors, auth failures, network timeouts, and validation errors. Agents consuming the MCP server get structured errors they can reason about, not raw stack traces.

Integrate inline Semgrep scanning as build gate

Add the Bob Semgrep scan step after code generation. Define the rule set: no hardcoded credentials, no unsafe deserialization, no uncaught exceptions leaking internals. Generation fails if any rule fires — the user sees the issue in Bob's Findings panel before any code is saved.

✓

Milestone 2 test gate

  • Sending a malformed parameter returns a structured validation error, not a crash
  • No API keys appear anywhere in the generated source files — only env var references
  • Semgrep scan passes cleanly on all generated code
  • A deliberately injected secret in the spec is caught and rejected by secrets detection
M3
Governance layer — Week 2

Add audit logging and human approval checkpoints

Days 8–12

Add the governance features that turn this from a developer tool into an enterprise-ready platform. Audit logging means every tool invocation is traceable. Approval checkpoints mean humans stay in the loop before code ships.

Generate audit logging into every tool handler

Every generated tool handler must log: timestamp, tool name, caller identity (if available), sanitized input parameters (no secrets), response status, and latency. Log format is structured JSON, ready for ingestion into any SIEM. This is a generation-time decision, not a runtime config.

Implement Bob approval checkpoint between plan and code

Surface the structured plan artifact to the user before Builder mode runs. The user sees: list of endpoints being implemented, auth methods detected, rate limit strategies proposed, and any ambiguities flagged. User must explicitly approve or modify before generation proceeds.

Implement second approval checkpoint before ship

After Tester mode completes, surface the test results and Findings panel output. User approves the server as deployable. This gate is baked into the BobShell recipe — it cannot be skipped without modifying the recipe explicitly.

Add BobShell execution trace to output package

Every generated server package includes a trace file — a complete log of every Bob agent action, decision, and model call that produced it. This is the provenance record. Enterprises can reproduce the generation exactly from this trace.

✓

Milestone 3 test gate

  • Every tool invocation produces a structured audit log entry with no secrets present
  • Pipeline halts at the approval checkpoint and does not proceed without explicit user input
  • The trace file in the output package allows exact reproduction of the generation run
  • Skipping the approval gate via CLI flag is explicitly blocked by the recipe
M4
Testing layer — Week 2–3

Adversarial test suite generation and Bob Tester mode

Days 11–16

Build the automated test generation pipeline. This is what separates the Forge from every existing OpenAPI-to-code tool. Not just unit tests — an adversarial suite that tries to break the generated server the way a misbehaving agent would.

Generate happy-path unit tests for every tool

For each generated tool, create a unit test using valid inputs drawn from the spec's example values or generated from the schema. Tests run against the generated server with a mock HTTP backend. All happy-path tests must pass before the adversarial suite runs.

Generate malformed input tests

For each parameter in every tool, generate edge case inputs: wrong type, missing required field, value out of range, empty string where string is required, null where null is not allowed. The server must return a structured validation error for every case — never a crash or an unhandled exception.

Generate injection attempt tests

Test prompt injection strings, SQL injection patterns, and shell metacharacters in all string parameters. The generated server must sanitize or reject these inputs before they reach the HTTP layer. Results surface in Bob's Findings panel with severity ratings.

Generate rate limit and auth failure tests

Simulate 429 and 401/403 responses from the upstream API. The generated server must handle these with structured error responses and sensible retry behavior — not raw HTTP errors passed through to the calling agent.

✓

Milestone 4 test gate

  • All happy-path tests pass with zero errors
  • Every malformed input returns a structured error — no unhandled exceptions in logs
  • All injection attempt tests return rejection responses — nothing reaches the HTTP backend
  • Rate limit simulation returns structured retry guidance, not a raw 429
  • Test suite itself is included in the output package and re-runnable by the user
M5
Documentation layer — Week 3

Bob Documenter mode and complete output package

Days 14–18

The output of the Forge is not just code — it is a complete, deployable package that a developer can hand to a junior engineer or a DevOps pipeline without any additional context. This milestone builds that package.

Generate README with getting-started guide

Auto-generated README covering: what the server does, which API it wraps, how to set environment variables, how to run locally, how to run the test suite, and how to deploy. Generated from the spec metadata and the plan artifact — not a generic template.

Generate environment variable reference

A complete table of every env var the server needs, with description, format, example value, and whether it is required or optional. Sourced directly from the auth schemes detected in the spec by Architect mode.

Generate tool reference documentation

For every MCP tool in the generated server: name, description, input schema, output schema, example call, and known error conditions. This is what the agent developer reads to understand what tools are available and how to use them.

Package the BobShell recipe as a first-class artifact

The recipe file is included in the output package alongside the code. It references the exact spec hash, Forge version, and configuration used. Running the recipe with a new spec version produces an updated server with a clean diff against the previous output.

✓

Milestone 5 test gate

  • A developer with no prior context can follow the README to get the server running in under 5 minutes
  • Re-running the recipe on a modified spec produces a diff-able updated server with no manual edits needed
  • Every env var in the reference table is actually used in the generated code — no orphans, no missing entries
M6
Multi-model optimization — Week 3–4

Wire Bob multi-model routing for cost efficiency

Days 17–21

Replace the single-model default with a cost-optimized routing strategy. Spec parsing does not need a frontier model. Security review does. This milestone makes the Forge economically viable at scale.

Profile each pipeline stage for model requirements

Run all five stages on three different specs and measure: accuracy of output, tokens consumed, and latency per model tier (Granite, Mistral, Claude). Document the minimum model needed per stage to produce acceptable output.

Implement Bob multi-model routing rules

Spec ingestion and plan generation: Granite model. Code generation and security review: Claude Sonnet or equivalent. Adversarial test generation: Claude with extended reasoning. Documentation: Mistral or Granite. Wire the routing rules into the Orchestrator mode configuration.

Add cost estimation to the approval checkpoint

At the plan approval gate, show the user an estimated token cost for the full generation run before they approve. This uses the Bobcoin pricing model. Enterprise users need cost predictability before triggering generation on large specs.

✓

Milestone 6 test gate

  • Total Bobcoin cost per generation run is at least 30% lower than single-model baseline
  • Output quality is equivalent — the same test gate suite from M4 passes on routing-optimized output
  • Cost estimate shown at approval checkpoint is within 15% of actual cost post-run

Post-hackathon startup layer — Month 2+

Compliance adapter, recipe library, and provenance system

Weeks 5–12

The hackathon milestone proves the concept. This milestone builds the startup moat. Domain-specific compliance adapters, the public recipe library, and cryptographic provenance tracking are the three features that separate the Forge from any weekend competitor.

Build the financial services compliance adapter

Domain adapter for AML/transaction monitoring APIs: mandatory audit logging in the format required for financial regulators, PII detection and masking before data leaves the server, rate limit handling that respects central bank API quotas, and structured error codes that map to regulatory incident categories. Your direct AML experience at ING makes this faster and more accurate than any generic team could build it.

Launch public recipe library on GitHub

Publish pre-validated BobShell recipes for Stripe, Salesforce, GitHub, Jira, and HubSpot. Each recipe includes the full generation config, the test suite, the output server, and a README. The library is open source and community-contributable. Seed it with five recipes and watch the network effect start.

Implement cryptographic provenance tracking

Every generated server package gets a signed manifest: SHA-256 of the input spec, Forge version, Bob version, model routing config, and generation timestamp. The manifest is verifiable — enterprises can prove to auditors exactly what generated their MCP server. Store manifests in a public transparency log.

Build the CI/CD integration for automatic regeneration

GitHub Action and GitLab CI template that watches the upstream OpenAPI spec file in a repo. When the spec changes, the Action runs the Forge recipe automatically, opens a PR with the updated server, and attaches the test results and diff as PR comments. Once merged, the team's MCP server is always in sync with the upstream API.
